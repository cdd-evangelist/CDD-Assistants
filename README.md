# CDD-Assistants

## これは何？

CDD (Chat/Character/Chart Driven Development) 手法に基づいて、ユーザーの
「これを作りたい」を支援するMCPサービス群です。Claude Code での利用を想定しています。

## どんなことができるの？

CDD では、詳細な設計文書を作成し、それをもとにAIが実装を行うという手法で開発を進めます。

CDDアシスタントは、Plannerが設計文書の作成の支援を行い、BuilderがPlannerとの共同作業によって出来上がった
設計文書に基づいて、要望に最適な実装を行います。

## Vibe コーディングとの違いは？

Vibeコーディングの手法では、ユーザーが「こういうのを作りたい」っていうと
AIがちょっと提案してくれた後に、一気に実装まで進んで「できました」って、（一応）動くツールが出来上がります。
出来上がったものがあなたにちょうどぴったりであれば、それでOKですが、「ここはちょっとこう動いてほしい」
とか「この機能、どうやってできてるんだろう？」みたいな疑問が生じた際に、調べるのが難しい実装だったりします。

このアシスタントを利用すると、Plannerがユーザーが納得するまで設計を詰めるお手伝いをしてくれます。
また、Builderは、設計文書をもとに実装を行い、実装後に正しく設計通りに実装されているかどうかのチェックまで行います。
その工程は、対話（Chat）を通して行われるので、成果物に対する納得感が違うのではないかと思います。

## 想定される利用者は？

コーディングに不慣れな非エンジニアや、初級〜中級プログラマの方を対象としています。
設計文書よりもコードを読むほうが理解が早い、上級エンジニアの方には不向きかもしれません。。。

## こんな便利な効果もあります

設計文書をもとに実装を行う副次的効果で、非常に可読性の高いコードが実装される傾向にあります。
実装フェーズに移る際にプラットフォームを指定することで、同一設計でクロスプラットフォーム対応が容易になります。
また、設計文書がきれいに残るので、後から「ここどうなってるんだっけ？」を調べやすく、メンテナンスも楽になります。

## 全体の流れ

```
「こういうのが欲しい」        設計文書群            実装コード
  （曖昧なイメージ）              ↓                    ↑
        ↓                       ↓                    |
    【Planner】      →     【Builder】      →    実装エージェント
    壁打ち・設計支援        分割・実装指示
```

| パッケージ | 役割 |
|-----------|------|
| `planner/` | 構想の言語化・設計文書の作成支援 |
| `builder/` | 設計文書の分析・チャンク分割・実装レシピ生成・実行 |
| `docs/` | 設計文書・ロードマップ |

## Planner — 壁打ち・設計支援

「なんかこういう感じのが欲しいんだけど…」というモヤモヤを、設計文書に落とし込むまでをお手伝いします。

### 壁打ちの流れ

```
Phase 0: モヤモヤを言葉にする — 「なんかこういう感じのが欲しいんだけど…」
Phase 1: 構想の言語化       — 核心は何か？ 既存との違いは？ ユーザーは誰か？
Phase 2: 設計の切り口を選ぶ — ユースケース / アーキテクチャ / データモデル ...
Phase 3: 文書を書きながら決める — 書くことで曖昧さが見つかる
Phase 4: 文書間の整合性を取る — 用語・命名・参照の統一
Phase 5: 完了判定           — Builder に渡せる状態か
```

ちなみに、ちょっとしたスクリプトや「やっつけでいいから早く動くのが欲しい」という場合は、
Planner が自動で判断して、CDD のフルコースをスキップします（コンシェルジュ機能）。

### MCP ツール

| ツール | 概要 |
|--------|------|
| `clarify_idea` | 曖昧な構想を4軸（対象ユーザー・価値・スコープ・制約）で分析。ワンショット/フルCDD の判定も行う |
| `suggest_approach` | 設計の切り口（ユースケース駆動、データモデル駆動等）を提案 |
| `design_context` | 設計文書群をスキャンし、進捗・参照関係・未決事項のスナップショットを返す |
| `track_decision` | 壁打ち中の決定事項を構造化記録 |
| `check_consistency` | 設計文書間の整合性を5カテゴリでチェック |
| `check_readiness` | Builder に渡せる状態かを判定 |

## Builder — 設計から実装へ

設計文書が完成したら、Builder が自動で実装を進めます。
設計文書を分析してチャンク（実装の塊）に分割し、依存関係を考慮した順序で実装を実行します。

### パイプライン

```
設計文書群 → analyze_design → split_chunks → export_recipe → 実装レシピ
                                                                 ↓
                                                          実装エージェントが
                                                          チャンク単位で実装
                                                                 ↓
                                                          リファレンス照合
                                                          （設計通りか検証）
```

実装後に「ラウンドトリップ検証」を行い、設計文書と実装の間に矛盾がないかを自動チェックします。
コードの中身を読まなくても、日本語同士の照合で設計意図の逸脱を検出できます。

### MCP ツール

| ツール | 概要 |
|--------|------|
| `analyze_design` | 文書構造の分析、レイヤー分類、依存グラフ構築 |
| `split_chunks` | 依存グラフのトポロジカルソートに基づくチャンク分割 |
| `validate_refs` | wiki-link 整合性・UC/AC 欠番検出 |
| `export_recipe` | チャンク群を自己完結型の実装レシピとして出力 |
| `load_recipe` | レシピファイルを読み込み、実行状態を初期化 |
| `next_chunks` | 依存が解決済みのチャンクを返す |
| `complete_chunk` | チャンクの完了を検証し、記録 |
| `execution_status` | 全体の実行進捗を可視化 |

## セットアップ

### 前提条件

- Node.js 18 以上
- [Claude Code](https://docs.anthropic.com/en/docs/claude-code) がインストール済みであること

### インストール

```bash
git clone https://github.com/cdd-evangelist/CDD-Assistants.git
cd CDD-Assistants
./setup.sh            # 両方セットアップ
./setup.sh planner    # Planner のみ
./setup.sh builder    # Builder のみ
```

### Claude Code への MCP 登録

ビルド後、以下のコマンドで Claude Code に MCP サーバーを自動登録できます:

```bash
./setup.sh mcp-install        # 現在のプロジェクトに登録
./setup.sh mcp-install --all  # 全プロジェクトに一括登録
```

登録後、Claude Code を再起動すると反映されます。

登録を解除するには:

```bash
./setup.sh mcp-uninstall        # 現在のプロジェクトから解除
./setup.sh mcp-uninstall --all  # 全プロジェクトから一括解除
```
