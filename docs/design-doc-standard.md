# 設計文書標準

更新日: 2026-04-11

## 1. 目的

CDD で扱う設計文書のフォーマットと分割粒度を標準化する。

設計文書とコードは**デジタルツイン**の関係にある。表現型が異なるだけで、記述される内容は一致しているべきであり、乖離が生じた場合はラウンドトリップ検証で検出する。

この標準が対象とするのは:

- CDD-Assistants（Planner / Builder）が処理する設計文書
- CDD を利用するプロジェクトの設計文書

## 2. 文書階層と分割粒度

### 2.1 3階層モデル

| 階層 | 読者 | 内容 | 分割単位 |
|---|---|---|---|
| **基本設計** | 人間 | 目的・スコープ・アーキテクチャ・技術選定・設計原則 | システム全体で1本 |
| **機能設計** | 人間 + AI | 「何を・なぜ」— 機能の概要、入出力、制約、他機能との関係 | 人間が承認判断する単位 |
| **詳細設計** | AI | 「どう動くか」— API 定義、データ構造、処理フロー、エラー処理 | 実装コードと対応が取れる単位 |

加えて、**ユースケース**をアクター別（人間向け / AI 向けなど）に分離する。

### 2.2 階層ごとの特徴

**基本設計（BasicDesign.md）**

- システム全体を俯瞰するための文書
- 人間が「このシステムは何をするものか」を理解できることが目的
- サイズ上限は設けない（ただし簡潔さを重視する）

**機能設計**

- 人間が仕様の妥当性を判断し、承認するための文書
- 「この機能は何か」「なぜ必要か」「入出力の概要」「受け入れ基準」を記述する
- 分割の境界は人間にとって自然な機能単位
- サイズ上限は設けない

**詳細設計**

- AI（Builder）が読んで実装に落とせる精度を持つ文書
- 実装コードの構造と対応関係を維持する
- 1ファイルあたり **500行 / 5,000トークン以下** を推奨（Builder の source_docs 制約との整合）

### 2.3 設計文書とコードの対応関係

詳細設計ファイルは、実装コードとの対応が追えるように分割する。

```
詳細設計ファイル       ←→  実装コード
──────────────────────────────────────
mcp-tools.md          ←→  src/mcp/*.ts
notebook-spec.md      ←→  src/notebook.ts
execution-engine.md   ←→  src/engine/*.ts
```

この対応関係により:

- Builder がどの設計文書からどのコードを生成するかが明確になる
- ラウンドトリップ検証で乖離を局所的に検出できる
- 設計変更の影響範囲が限定される

## 3. ユースケース文書

アクターごとに分離する。

| ファイル | 内容 |
|---|---|
| `user-usecases.md` | 人間のユーザーが行う操作シナリオ |
| `ai-side-usecases.md` | AI エージェントが行う操作シナリオ |

ユースケースは機能設計・詳細設計から参照される。以下の ID 体系で識別する:

- **UC-ID**（Use Case ID）: ユースケースの識別子。`UC-1`, `UC-2`, ... の連番
- **AC-ID**（Acceptance Criteria ID）: 受け入れ基準の識別子。`AC-1`, `AC-2`, ... の連番

欠番検出は validate_refs で行う。

## 4. サイズガイドライン

| 階層 | 推奨上限 | 根拠 |
|---|---|---|
| 基本設計 | なし | 人間が俯瞰する文書。簡潔さは求めるが制限はしない |
| 機能設計 | なし | 人間の承認単位。必要な情報を過不足なく記述する |
| 詳細設計 | 500行 / 5,000トークン | Builder の source_docs 制約（1〜2本、8k トークン以内）に収まること |
| ユースケース | なし | アクター単位の分割で十分 |

詳細設計がサイズ上限を超える場合は、処理単位やサブモジュール単位でさらに分割する。

詳細設計のサイズ上限は、LLM が長いコンテキストの中間部分への注意が低下する「Lost in the Middle」問題への対策でもある。

## 5. フォルダ構成

設計文書が増えてもフラットに並べない。階層ごとにフォルダを分け、開いた瞬間に全体像がわかるようにする。

### 5.1 推奨構成

```
docs/
├── README.md                    ← 文書一覧・導線
├── basic-design.md              ← 基本設計（1本）
├── usecases/                    ← ユースケース（アクター別）
│   ├── user-usecases.md
│   └── ai-side-usecases.md
├── features/                    ← 機能設計（人間が承認する単位）
│   ├── {機能名}.md
│   └── ...
├── details/                     ← 詳細設計（AI が実装する単位）
│   ├── {モジュール名}.md
│   └── ...
└── ref/                         ← リファレンス（検証結果、補足資料）
    └── ...
```

### 5.2 フォルダの役割

| フォルダ | 内容 | 対応する階層 |
|---|---|---|
| `docs/` (ルート) | basic-design.md と README.md | 基本設計 |
| `usecases/` | アクター別のユースケース文書 | ユースケース |
| `features/` | 機能単位の設計文書 | 機能設計 |
| `details/` | 実装コードと対応する詳細仕様 | 詳細設計 |
| `ref/` | リファレンス、検証結果、補足資料 | — |

### 5.3 docs/README.md

`docs/README.md` は文書一覧と導線を提供する。フォルダを開いた人が迷わず目的の文書にたどり着けることが目的。

```markdown
# {プロジェクト名} 設計文書

## 基本設計
- [基本設計書](basic-design.md)

## ユースケース
- [ユーザーユースケース](usecases/user-usecases.md)
- [AI サイドユースケース](usecases/ai-side-usecases.md)

## 機能設計
- [{機能名}](features/{機能名}.md)

## 詳細設計
- [{モジュール名}](details/{モジュール名}.md)
```

### 5.4 複数コンポーネントを含むプロジェクト

プロジェクト内に独立したコンポーネント（別プロセス、別サーバー、明確に異なる責務を持つモジュール等）が複数ある場合は、コンポーネント単位でフォルダを分け、その中に3階層モデルを展開する。

```
docs/
├── README.md                        ← プロジェクト全体の導線
├── design-doc-standard.md           ← プロジェクト横断の標準・方針
├── roadmap.md
├── planner/
│   ├── basic-design.md
│   ├── usecases/
│   ├── features/
│   ├── details/
│   └── ref/
└── builder/
    ├── basic-design.md
    ├── usecases/
    ├── features/
    ├── details/
    └── ref/
```

**判断基準:** コンポーネント分割するのは、以下のいずれかに該当する場合:

- 独立してビルド・デプロイできる（別プロセス、別パッケージ）
- 責務が明確に異なり、設計文書の読者が異なる
- それぞれに固有のユースケース・データモデルがある

該当しなければ §5.1 のフラットな構成で十分。

## 6. ファイル命名規則

### 6.1 ケースの選択

プロジェクト開始時にファイル名のケースを選択し、プロジェクト内で統一する。

| ケース | 例 | 主な文化圏 |
|---|---|---|
| kebab-case | `chunk-execution.md` | Web / Node.js / Linux |
| PascalCase | `ChunkExecution.md` | C# / .NET |
| snake_case | `chunk_execution.md` | Python / Ruby |

どのケースを選んでもよいが、1プロジェクト内で混在させない。

### 6.2 階層ごとの命名パターン

| 階層 | 命名規則 | 例（kebab-case） |
|---|---|---|
| 基本設計 | `basic-design.md` 固定 | `basic-design.md` |
| ユースケース | `{アクター}-usecases.md` | `user-usecases.md` |
| 機能設計 | `{機能名}.md` | `chunk-execution.md` |
| 詳細設計 | `{モジュール名}.md` | `mcp-tools.md` |
| README | `README.md` 固定 | `README.md` |

### 6.3 命名の指針

- フォルダで階層が分かれるため、`-design` や `-spec` のサフィックスは不要（`features/notebook.md` で十分）
- 基本設計と README はプロジェクト間で固定名にする（探しやすさのため）
- 英語を推奨する（日本語ファイル名はシェルやツールとの相性が悪い）

## 7. 記述の品質ルール

設計文書は「完成された文書」として読めること。作業ログや編集痕跡を残さない。

### 7.1 禁止する表現

| 禁止パターン | 例 | 理由 |
|---|---|---|
| 修正・追記の注釈 | 「以下を追加しました」「ここを修正」 | 文書は現在の仕様を表す。変更履歴は Git に任せる |
| 作業ログ的な記述 | 「前回の議論を踏まえ〜」「Issue #N で指摘された〜」 | 文書の自己完結性が損なわれる |
| 仮置きの注釈 | 「TODO: あとで書く」「TBD」 | draft ステータスであっても、書けるところは書き切る |
| 過剰な経緯説明 | 「当初は A 案だったが B 案に変更した」 | 設計判断セクションに書くなら可。本文中には不要 |

### 7.2 推奨する書き方

- **現在形で書く** — 「〜する」「〜である」。「〜に変更した」「〜を追加した」ではない
- **文書単体で完結させる** — 他の文書や Issue を読まないと意味が通らない記述を避ける。参照する場合はリンクと要約を添える
- **セクションの役割を守る** — テンプレートの各セクションが求める内容だけを書く。関係ない情報を混ぜない

### 7.3 変更履歴の管理

文書本文に変更履歴を書かない。以下の手段で管理する:

- **CHANGELOG.md** — 全設計文書の変更を日付・コミットハッシュ付きで記録する（後述 §7.4）
- **フロントマターの更新日** — 各文書の最終更新日だけを記録する
- **Git** — ファイル単位の詳細な変更履歴は `git log` で追跡する
- **設計判断セクション** — 「なぜこの方式を選んだか」は設計判断として記録する。ただし経緯ではなく結論と理由を書く

### 7.4 CHANGELOG.md

`docs/CHANGELOG.md`（複数コンポーネントの場合は各コンポーネントフォルダ直下）に、設計文書の変更履歴を集約する。

```markdown
# 設計文書 変更履歴

## YYYY-MM-DD

- {ファイル名}: {変更内容の要約}（{コミットハッシュ}）
```

**運用ルール:**
- 設計文書を更新するたびに CHANGELOG.md にも1行追記する
- Planner / Builder が機械的に追記することを前提とする（人間が手動で管理しない）
- 各エントリにはコミットハッシュを付け、詳細は `git show` で参照できるようにする

## 8. テンプレート

### 8.1 基本設計テンプレート

```markdown
# {プロジェクト名} 基本設計書

更新日: YYYY-MM-DD

## 1. プロジェクト概要

### 1.1 目的
（このシステムは何か、どんな課題を解決するか）

### 1.2 スコープ
（やること・やらないこと、既存システムとの棲み分け）

## 2. アーキテクチャ

### 2.1 全体構成
（コンポーネント図、データフロー概要）

### 2.2 データの所在
（Source of Truth がどこか、データの流れ）

## 3. 技術選定

（技術スタック、選定理由）

## 4. データモデル
<!-- 該当する場合 -->

（ER図、テーブル定義、主要クエリ）

## 5. 制約・前提

（技術的制約、運用上の前提条件）

## 6. 今後の検討事項
<!-- 任意 -->

（未決定事項、将来の拡張）

## 関連ドキュメント

（機能設計・詳細設計・ユースケースへのリンク）
```

**記述の指針:**
- 人間が「このシステムは何をするものか」を理解できることが最優先
- 技術選定には「なぜその技術を選んだか」の理由を必ず添える
- 関連ドキュメントで、機能設計・詳細設計への導線を示す

### 8.2 機能設計テンプレート

ICONIX プロセスのロバストネス分析を参考に、ユースケースと詳細設計の間の曖昧さを除去する役割を持つ。

```markdown
# {機能名} 機能設計書

更新日: YYYY-MM-DD

## 1. 概要

（この機能は何か、なぜ必要か）

## 2. 構成要素

### 2.1 バウンダリ（外部との接点）
（ユーザーや外部システムとやり取りする境界。画面、API、CLI、MCP ツールなど）

### 2.2 エンティティ（扱うデータ）
（この機能が読み書きするデータの種類と構造。テーブル、設定ファイル、メッセージ形式など）

### 2.3 コントローラー（主要な処理）
（バウンダリから受け取った入力を、エンティティを使って処理する流れ。「○○を検証する」「○○を生成する」のように動詞で記述する）

## 3. ユースケース

（UC-ID / AC-ID で識別。機能に関連するユースケースと受け入れ基準を列挙または参照）

## 4. 受け入れ基準

EARS 記法で記述する:
- When [条件], the system shall [振る舞い]
- While [状態], the system shall [振る舞い]
- Where [制約], the system shall [振る舞い]

## 5. 設計判断

（なぜこの方式を選んだか、検討した代替案との比較）

## 6. 検証方針

（ラウンドトリップ検証の観点、受け入れ基準の検証方法）

## 関連ドキュメント

（基本設計・詳細設計へのリンク）
```

**記述の指針:**
- §2 の構成要素分析で、ユースケースの自然言語から曖昧さを除去する
- §4 の受け入れ基準は EARS 記法で機械可読性を高める（ただし全項目に強制はしない。自然言語のほうが明確な場合はそちらを使う）
- 人間がこの文書を読んで「仕様として妥当か」を承認判断できることが目的

### 8.3 詳細設計テンプレート

AI（Builder）が読んで実装に落とせる精度を持つ文書。実装コードとの対応関係を維持する。

```markdown
# {対象モジュール名} 詳細設計書

更新日: YYYY-MM-DD
対応コード: src/{path}

## 1. 概要

（この詳細設計の対象範囲、対応する機能設計へのリンク）

## 2. 一覧

（API / 関数 / ツールのサマリテーブル）

| 名前 | 種別 | 概要 |
|---|---|---|
| ... | ... | ... |

## 3. 詳細仕様

### 3.N {API名 / 関数名}

**入力:**
（パラメータ、型、制約）

**出力:**
（戻り値、型）

**処理ステップ:**
1. ...
2. ...

**エラー処理:**
（異常系の振る舞い）

## 4. 横断的関心事
<!-- 該当する場合 -->

（パフォーマンス、セキュリティ、トークンコスト等）

## 関連ドキュメント

（機能設計へのリンク）
```

**記述の指針:**
- ヘッダの「対応コード」で、この文書がどの実装コードのツインかを明示する
- 処理ステップは AI が実装できる精度で書く。曖昧な表現（「適切に処理する」等）は避ける
- 500行 / 5,000トークンを超える場合はファイルを分割する

### 8.4 ユースケーステンプレート

```markdown
# {アクター名} ユースケース

更新日: YYYY-MM-DD

## UC-1/AC-1: {ユースケース名}

**アクター:** {誰が}
**目的:** {何のために}
**事前条件:** {前提}

### 基本コース（正常系の主な流れ）
1. ...
2. ...

### 代替コース（異常系・分岐）
- {条件}: ...

### 受け入れ基準
- AC-1: ...
- AC-2: ...

### 図
<!-- mermaid のコードブロックでシーケンス図、フローチャート等を記述する -->
```

> **mermaid 記述例（テンプレート外）:**
> ```mermaid
> sequenceDiagram
>     actor User
>     User->>System: 操作
>     System-->>User: 結果
> ```

**記述の指針:**
- アクターごとにファイルを分離する（user-usecases.md, ai-side-usecases.md 等）
- UC-ID / AC-ID は連番で管理し、欠番は validate_refs で検出する
- 受け入れ基準は EARS 記法を推奨するが、必須ではない
- 図は mermaid で記述する。シーケンス図やフローチャートなど、文章だけでは伝わりにくい流れはなるべく図にする

### 8.5 適用の判断基準

全ての変更にこのテンプレートを適用する必要はない。問題サイズに応じて判断する:

| 変更の規模 | 適用レベル | 例 |
|---|---|---|
| 新機能 / 大規模改修 | 全階層（基本設計更新 + 機能設計 + 詳細設計） | 新コンポーネント追加 |
| 既存機能の拡張 | 機能設計更新 + 詳細設計更新 | ツール追加、仕様変更 |
| バグ修正 / 小改善 | 詳細設計の該当箇所を更新（または省略） | エラー処理修正 |
