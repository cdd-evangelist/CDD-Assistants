# CDD-Planner 設計書

## 1. 概要

CDD-Planner は、人の曖昧な構想を設計文書に落とし込むプロセスを支援する MCP サーバー。
CDD（Character Driven Development）の「コードを見て考えるな。日本語を見て考えろ」を実践するための **設計壁打ちパートナー**。

人が「こういうのが欲しい」と言ったとき、それを聞き出し、整理し、設計に変換する。
AI-Ghost-Shell の設計で人と Claude が 14本の設計文書を作り上げた、あのプロセス全体を構造化・再現可能にする。

### この設計が生まれた原体験

AI チャットに一言お願いしたら、いきなり実装が始まった。
確認もなく、設計もなく、コードが流れてくる。速い。でも——

**「ちょっと待って。そのスピードで動かれると、ついていけない。」**

自分のものを作っているのに、自分が蚊帳の外にされる感覚。
AI が勝手に走り出すと、人は置いてけぼりにされる。

この違和感を捨てずに「じゃあどうすればいいか」を突き詰めた結果が CDD であり、この Planner の設計に至った。
人が AI と一緒にものを作るとき、最も大切なのはスピードではなく**対話**だった。

### 解決する問題

人は「こういうものが欲しい」というイメージは持っている。
しかしそれをシステムの形に落とし込める人は少ない。

**誰しもが明確な指示を AI に出せるわけではない。**

素の LLM に曖昧な構想をぶつけると:
- いきなりコードを書き始める（設計を飛ばす）
- 聞こえのいい提案をするが、整合性がない
- セッションが長くなると過去の決定事項を忘れる
- 「何が決まって、何が未決か」が誰にも分からなくなる
- 完了条件がなく、いつ実装に移るべきか判断できない

Planner は、曖昧な「こうしたい」を構造化された設計文書群に変換するプロセスを支援する。
「置いてけぼりにされない AI 開発」を、仕組みとして実現する。

### 位置づけ

```
  「こういうのが欲しい」         設計文書群（完成品）
   （曖昧なイメージ）                  ↓
         ↓                           ↓
     【CDD-Planner】    →    【CDD-Builder】    →    実装コード（テスト済み）
      壁打ち・設計支援         レシピエンジン
                               + 実行エンジン
                               + 実行アダプタ（差し替え可能）
```

Planner の出力 = Builder の入力。
Planner は「何を作るか」を固める。Builder は「どう作るか」を自律的に実行する。
Builder は実行アダプタを通じて実装を行う（デフォルト: Claude Code、差し替え: ローカル LLM 等）。
その間に立つのは常に人。Planner は人の判断を支援するが、代替はしない。

## 2. 設計思想

### 2.1 Planner が担うもの / 担わないもの

| Planner が担う | Planner が担わない |
|---------------|-------------------|
| 曖昧な構想を引き出し、具体化を支援する | 人の代わりに意思決定する |
| 設計の切り口を提案する | 実装方針を決める |
| 決定事項を構造化して記録する | コードを書く |
| 文書間の整合性を監視する | 設計文書をチャンク分割する（→ Builder） |
| 未決事項を可視化する | テストを書く |
| 完了判定を支援する | デプロイする |

### 2.2 壁打ちの典型的な流れ

AI-Ghost-Shell の設計過程から抽出した、壁打ちのパターン:

```
Phase 0: モヤモヤを言葉にする
  「なんかこういう感じのが欲しいんだけど…」
  → まだシステムの形になっていない。イメージや不満がある段階
  → Planner が質問で引き出す: 誰のため？ 何が嬉しい？ 既存の何が不満？

Phase 1: 構想の言語化
  「伺かのAI版を作りたい」
  → 核心は何か？ 既存との違いは？ ユーザーは誰か？
  → ここで初めて「プロジェクト」として認識できる形になる

Phase 2: 設計の切り口を選ぶ
  → ユースケース / アーキテクチャ / データモデル / ポリシー ...
  → どこから攻めるかで設計の質が変わる

Phase 3: 文書を書きながら決める
  → 書くことで曖昧さが見つかる
  → 壁打ちで方針を決め、文書に反映する

Phase 4: 文書間の整合性を取る
  → 新しい文書が既存の文書と矛盾していないか
  → 用語・テーブル名・設定キー名の統一

Phase 5: 完了判定
  → 全文書が ✅ になったか
  → 未決事項が残っていないか
  → Builder に渡せる状態か
```

Phase 0〜1 が Planner の最も重要な仕事。ここを飛ばすと、あとの全てが歪む。

**Planner の行動原則: 拡散→収束**
- 構想が曖昧なうちは選択肢を広げる（Phase 0〜1）
- 基本方針が決まったら、収束に向かって伴走する（Phase 2〜5）
- このリズムは各 Phase 内でも再帰的に起きる（文書を書きながらの壁打ちでも、まず広げてから絞る）

### 2.3 壁打ちの原則

CDD の設計壁打ちで実際に機能した対話のエッセンスを、Planner の振る舞い指針として定義する。
これは `clarify_idea` や `suggest_approach` が対話を組み立てるときの基本姿勢になる。

**1. 日本語で十分に言語化してから次に進む**
- コードや技術選定に飛びつかない
- 「何を作るか」「誰のためか」「なぜ作るか」が自然言語で明確になるまでは、次の Phase に進まない
- 曖昧なまま進むと、あとの全てが歪む

**2. 判断ポイントでは必ず人に返す**
- Planner が勝手に決めない。選択肢を整理して「どう思う？」と返す
- 人の判断を待つことは遅延ではなく、設計品質への投資
- 小さな確認を頻繁に挟むことで、大きな手戻りを防ぐ

**3. 提案は複数出し、選択は人に委ねる**
- 1つの正解を押し付けるのではなく、2〜3の選択肢をトレードオフ付きで提示する
- 「私の推しは○○」と意見を添えてもいい。ただし決めるのは人
- 選択肢の粒度は、人が判断できるレベルに揃える

**4. 人の一言を拾って設計を深める**
- 「欲を言うと…」「ついでに…」「あと気になるのは…」——こうした一言は要件の種
- 聞き流さずに拾い上げ、設計に反映するか確認する
- 人が言語化しきれていない意図を、質問で引き出す

**5. 壁打ちは苦行ではなく、対話の楽しさがある**
- 設計は義務ではなく、ものづくりの最も楽しい工程
- 「こういう方向もあるよ」「それ面白いね」——知的好奇心を共有する空気を作る
- 人が「もっと話したい」と思えるテンポとトーンで進める

これらの原則は Phase 0〜5 の全体を通じて適用される。
特に Phase 0（モヤモヤを言葉にする）では、原則 4・5 が決定的に重要になる。
人が安心して曖昧なアイデアを口にできる空気がなければ、良い設計は始まらない。

### Builder への接続——設計の精度が実装の品質を決める

Planner でしっかり設計を詰めておけば、Builder は可読性が高く、メンテナンスしやすい実装を出力する。
Vibe コーディング（曖昧な指示で AI に一任する方式）とは、ここで差がつく。

設計が曖昧なまま実装に入ると、AI は不足情報をその場で補う。
型定義を即興で作り、命名を独自に決め、「念のため」の抽象化を追加する。
動くコードは出てくるが、書いた本人（AI）しか意図がわからないコードになる。

Planner を通した設計では：
- **型が契約書になる** — types.ts が先に決まり、各モジュールはそれに従うだけ
- **テスト名が仕様書になる** — 「UC-N の欠番を検出する」のように、設計文書の仕様文がテストケースにそのまま降りてくる
- **過剰な抽象化が消える** — 何を作るか明確なので、判断の余地がなくなる

壁打ちに時間をかけるほど、実装はシンプルになる。
これが CDD の「設計は精密に、実装はシンプルに」の実体。

## 3. MCP ツール仕様

### 3.1 `clarify_idea`

曖昧な構想を受け取り、設計に進むための問いを生成する。Phase 0 の入口。

**入力:**
```json
{
  "raw_idea": "AIと話してると毎回同じ説明するのがダルい。覚えててほしい",
  "existing_context": null
}
```

**処理:**
1. 入力テキストから「やりたいこと」「不満」「イメージ」を正規表現で抽出
2. テンプレート質問（必ず確認する軸）から、キーワードマッチで未回答の項目を選出
3. 未回答の軸に対応するテンプレート質問を選択（各軸から先頭1問、最大4問）
4. 既知の類似プロダクト・アプローチがあればキーワードマッチで参考として提示（最大3件）

**テンプレート質問（必ず確認する軸）:**
| 軸 | 質問例 |
|----|-------|
| 対象ユーザー | 誰が使う？ 自分だけ？ チーム？ |
| 価値 | 何が嬉しい？ 今の何が不満？ |
| スコープ | 個人用？ 配布する？ |
| 制約 | 技術的な前提や縛りはある？ |

**出力:**
```json
{
  "understood": {
    "core_desire": "AI が過去の会話を記憶し、繰り返しの説明を不要にしたい",
    "pain_point": "セッションごとにコンテキストがリセットされる",
    "implied_scope": "会話履歴の永続化、文脈の自動復元"
  },
  "questions": [
    {
      "question": "覚えてほしいのは事実（名前、好み）？ それとも過去の議論の流れ？",
      "why": "記憶の粒度でアーキテクチャが大きく変わる"
    },
    {
      "question": "使う AI は Claude だけ？ 他の LLM でも使いたい？",
      "why": "特定 LLM 依存か汎用設計かの分岐点"
    },
    {
      "question": "自分だけで使う？ 他の人にも配りたい？",
      "why": "パッケージング・配布の要否"
    }
  ],
  "similar_approaches": [
    { "name": "mem0", "relevance": "LLM の記憶レイヤー。ただしホスト型" },
    { "name": "Claude の auto-memory", "relevance": "最も近いが、カスタマイズ性が低い" }
  ]
}
```

**Phase 0 → 1 の遷移判定:**

`clarify_idea` は毎回、テンプレート4軸の充足度をチェックする:

| 軸 | 状態 |
|----|------|
| 対象ユーザー | ✅ 回答済み / ❓ 未回答 |
| 価値 | ✅ / ❓ |
| スコープ | ✅ / ❓ |
| 制約 | ✅ / ❓ |

- 全軸 ✅ → 「`suggest_approach` に進めます」を提示
- 1つでも ❓ → Phase 0 を継続し、未回答の軸に焦点を当てた質問を出す

往復回数ではなく充足度で判断する。1往復で全部埋まれば即座に進む。

**拡散と収束:**

`clarify_idea` は4軸の充足度に応じてモードを切り替える:

```
充足度 低（0〜1軸）→ 拡散モード
  - 「こういう方向もありますよ」と選択肢を広く提示
  - 類似プロダクト、別のアプローチ、意外な切り口を見せる
  - 人の発想を刺激し、可能性を広げる

充足度 中（2〜3軸）→ 収束モード
  - 散らばったアイデアを4軸に束ねて整理する
  - 「つまりこういうことですか？」と要約して確認
  - 残りの未回答軸に焦点を絞った質問を出す

充足度 高（4軸）→ 遷移提案
  - 「suggest_approach に進めます」を提示
```

構想が曖昧なうちは広げる。方向が見えてきたら束ねる。
このリズムが、人が一人では辿り着けない設計に導く。

### 3.2 `design_context`

プロジェクトの設計状況を把握する。セッション開始時や、長い壁打ちの途中で呼ぶ。

**入力:**
```json
{
  "project_dir": "path/to/Documents/AI-Ghost-Shell/"
}
```

**処理:**
1. ディレクトリ内の .md ファイルを列挙
2. 各文書のフロントマター or 先頭セクションからメタデータを抽出
3. 決定事項（Decision）と未決事項（Open Question）を収集
4. 文書間の参照関係を構築
5. 進捗サマリーを生成

**出力:**
```json
{
  "project": "AI-Ghost-Shell",
  "documents": [
    {
      "path": "BasicDesign.md",
      "status": "complete",
      "summary": "DB スキーマ（5テーブル）、2エディション構成",
      "decisions": [
        "行番号ポインタで .jsonl を参照（DBにログは取り込まない）",
        "Git版 / Lite版の2エディション"
      ],
      "open_questions": []
    },
    {
      "path": "ghost-security.md",
      "status": "draft",
      "summary": "攻撃ベクトルと緩和策",
      "decisions": ["trust level は3段階"],
      "open_questions": ["サンドボックス実行の具体的な実装方式"]
    }
  ],
  "overall_progress": {
    "complete": 12,
    "draft": 1,
    "planned": 1,
    "total": 14,
    "readiness": "not_ready"
  },
  "dependency_graph": {
    "BasicDesign.md": [],
    "ghost-security.md": ["BasicDesign.md"]
  },
  "total_tokens": 28500,
  "unresolved_questions": [
    {
      "source": "ghost-security.md",
      "question": "サンドボックス実行の具体的な実装方式",
      "blocking": false
    }
  ]
}
```

### 3.3 `suggest_approach`

構想やアイデアに対して、設計の切り口を提案する。

**入力:**
```json
{
  "idea": "AIの人格設定を管理・配布するツールを作りたい",
  "context": "（既存の設計文書があれば、design_context の出力）",
  "constraints": ["Obsidian で文書を書く", "実装は Builder（サブエージェント）に渡す"]
}
```

**アプローチテンプレート:**

コアテンプレート（ほぼ全プロジェクトで提案）:

| 名前 | 攻め方 | 向いている場面 |
|------|--------|--------------|
| ユースケース駆動 | 誰が何をするかから攻める | 要件が曖昧な初期段階 |
| データモデル駆動 | 何を保存するかから攻める | データ構造が核心のシステム |
| インターフェース駆動 | 外から見た振る舞いから攻める | 操作体験が重要なツール |

拡張テンプレート（キーワードマッチでプロジェクトの性質に応じて追加提案）:

| 名前 | 攻め方 | 向いている場面 |
|------|--------|--------------|
| ポリシー駆動 | 何を許し何を禁じるかから攻める | 権限・制約が多いシステム |
| 脅威駆動 | セキュリティの観点から攻める | 外部入力を扱うシステム |
| 比較駆動 | 既存プロダクトとの差分から攻める | 類似サービスが存在する領域 |
| イベント駆動 | イベント・トリガーから攻める | リアクティブ・非同期処理が多いシステム |
| ワークフロー駆動 | 業務フロー・手順から攻める | 人間の作業フローを自動化するシステム |

**出力:**
```json
{
  "approaches": [
    {
      "name": "ユースケース駆動",
      "description": "誰が何をするかから攻める。機能の抜け漏れが出にくい",
      "source": "core",
      "suggested_documents": [
        "ユーザー側ユースケース一覧",
        "AI側ユースケース一覧"
      ],
      "good_for": "要件が曖昧な初期段階"
    },
    {
      "name": "ポリシー駆動",
      "description": "何を許し何を禁じるかから攻める。AI の振る舞い制御が核心なら有効",
      "source": "extended",
      "suggested_documents": [
        "ポリシー定義",
        "権限モデル"
      ],
      "good_for": "権限・制約が多いシステム"
    }
  ],
  "recommendation": "ユースケース駆動 → データモデル → ポリシーの順が効果的"
}
```

### 3.4 `track_decision`

壁打ち中の決定事項を記録する。文書への自動反映はしないが、後で文書を書く際の入力になる。

**記録の粒度:**
- 記録する: 他の文書に影響する方針決定（`affects` が1つ以上）
- 記録しない: 影響がその文書内で閉じる修正（見出し変更、サンプル修正等）

`track_decision` は **文書をまたぐ方針変更の専用ツール**。文書内の修正は普通に書き直せばいい。

**入力:**
```json
{
  "decision": "messages テーブルを廃止し、.jsonl を行番号ポインタで参照する",
  "rationale": "DB を軽量に保つため。ログは外部ファイルのままアクセスする",
  "affects": ["BasicDesign.md", "mcp-tools.md", "operation-flows.md"],
  "supersedes": "旧設計: messages テーブルにログを全件格納"
}
```

**処理:**
- プロジェクトの決定ログ（`decisions.jsonl`）に追記
- 影響する文書にフラグを立てる（要更新）

**出力:**
```json
{
  "decision_id": "DEC-012",
  "recorded_at": "2026-02-28T...",
  "affected_documents_status": [
    { "path": "BasicDesign.md", "needs_update": true },
    { "path": "mcp-tools.md", "needs_update": true },
    { "path": "operation-flows.md", "needs_update": true }
  ]
}
```

### 3.5 `check_consistency`

設計文書群の整合性をチェックする。新しい文書を書いた後や、大きな方針変更の後に呼ぶ。

**入力:**
```json
{
  "project_dir": "path/to/Documents/AI-Ghost-Shell/",
  "focus": ["terminology", "references", "coverage"]
}
```

**チェック項目:**

| カテゴリ | チェック内容 |
|---------|------------|
| terminology | 同じ概念に異なる名称が使われていないか（例: `episodes` vs `episode_memories`） |
| references | 文書間リンクの整合性、存在しないセクションへの参照 |
| coverage | ユースケースに対応する設計文書があるか、逆も |
| decisions | 決定ログと文書の内容が一致しているか |
| staleness | 決定後に更新されていない文書がないか |

**出力:**
```json
{
  "status": "warn",
  "issues": [
    {
      "category": "terminology",
      "severity": "warn",
      "message": "episode-extraction.md では 'episodes' と記載、BasicDesign.md では 'episode_memories'",
      "suggestion": "episode_memories に統一"
    },
    {
      "category": "staleness",
      "severity": "info",
      "message": "DEC-012 の影響で operation-flows.md が未更新",
      "suggestion": "Flow 2, 3 のシーケンス図を更新"
    }
  ],
  "summary": {
    "errors": 0,
    "warnings": 1,
    "info": 1
  }
}
```

### 3.6 `check_readiness`

設計文書群が Builder に渡せる状態かを判定する。

**入力:**
```json
{
  "project_dir": "path/to/Documents/AI-Ghost-Shell/",
  "required_coverage": ["usecases", "data_model", "interfaces", "policies"]
}
```

**チェック項目:**

| 基準 | 内容 |
|------|------|
| 文書完了 | 全文書の status が complete |
| 未決事項 | blocking な open_question が残っていない |
| 整合性 | check_consistency で error がゼロ |
| カバレッジ | 必要な設計領域が全て文書化されている |
| 技術選定 | 言語、フレームワーク、ディレクトリ構成が決まっている |

**出力:**
```json
{
  "ready": false,
  "blockers": [
    {
      "type": "missing_coverage",
      "message": "技術選定（言語・フレームワーク）が未決",
      "suggestion": "実装技術を決定し、tech-stack.md を作成"
    }
  ],
  "warnings": [
    {
      "type": "consistency",
      "message": "用語の揺れが1件"
    }
  ],
  "handoff_summary": "14文書中13完了、ブロッカー1件を解消すれば Builder に渡せる"
}
```

## 4. データモデル

### 4.1 決定ログ: `decisions.jsonl`

プロジェクトディレクトリに自動生成。壁打ちの記録が蓄積される。

```jsonl
{"id":"DEC-001","decision":"2エディション構成（Git版+Lite版）","rationale":"...","affects":["BasicDesign.md"],"created_at":"2026-02-27T...","supersedes":null}
{"id":"DEC-002","decision":"messages テーブル廃止","rationale":"...","affects":["BasicDesign.md","mcp-tools.md"],"created_at":"2026-02-28T...","supersedes":"旧: messages テーブル"}
```

### 4.2 文書メタデータの埋め込み

各設計文書の先頭にフロントマターを付与（Obsidian 互換）:

```yaml
---
status: complete          # draft | in_progress | complete
layer: foundation         # foundation | specification | usecase | interface | execution | context
last_reviewed: 2026-03-01
decisions: [DEC-001, DEC-003, DEC-005]
open_questions: []
---
```

Planner は既存文書にフロントマターがなくても動作する（本文から推定）。
フロントマターがあれば、より正確な状況把握が可能。

**Obsidian 互換ルール:**
- Planner のカスタムキー（`status`, `layer`, `decisions` 等）は Obsidian の予約キーと衝突しない（検証済み）
- `tags`: **削除禁止、追加は必要に応じて**（重複チェックしてから追加）
- `aliases`, `cssclasses`: Planner は触らない
- キーの書き出し順序は `tags` → Planner キーの順で制御する（実装時）

## 5. Planner と Builder の接続

```
Planner                          Builder
───────                          ───────
clarify_idea          →（構想の引き出し）
suggest_approach      →（切り口提案）
design_context        →（状況把握）
track_decision        →（決定記録）
check_consistency     →（整合性）
check_readiness       →（完了判定）
                      ↓
              「ready: true」
                      ↓
                                 ┌─ レシピエンジン ─────────────┐
                                 │ analyze_design →（構造分析） │
                                 │ split_chunks   →（分割）     │
                                 │ validate_refs  →（参照検証） │
                                 │ export_recipe  →（レシピ出力）│
                                 └──────────────────────────────┘
                                                ↓
                                          recipe.json
                                                ↓
                                 ┌─ 実行エンジン ──────────────┐
                                 │ load_recipe       →（初期化）│
                                 │ next_chunks       →（次取得）│
                                 │ complete_chunk    →（検証）  │
                                 │ execution_status  →（進捗）  │
                                 └──────────────────────────────┘
                                                ↓
                                        実行アダプタ
                                   （claude-code / local-llm）
                                                ↓
                                        実装コード（テスト済み）
```

Planner の `check_readiness` が `ready: true` を返した時点で、
Builder にプロジェクトディレクトリを渡す。

Builder は Planner が付与したフロントマター・決定ログ（`decisions.jsonl`）を活用して、
より正確な分割を行う。具体的には:
- フロントマターの `status`, `layer` → 文書のレイヤー分類・完了状態の判定に使用
- フロントマターの `decisions`, `open_questions` → 依存グラフの補強に使用
- `decisions.jsonl` → 決定事項と影響文書の関係を依存分析に反映

## 6. AI-Ghost-Shell での振り返り

実際にこのセッション群で行った壁打ちを、Planner のツールに当てはめると:

| 実際にやったこと | 対応ツール |
|----------------|-----------|
| 「AIに記憶を持たせたい、伺かみたいな…」という曖昧な構想から始めた | `clarify_idea` |

すべて手動でやっていたことを、ツールで構造化・自動化する。
