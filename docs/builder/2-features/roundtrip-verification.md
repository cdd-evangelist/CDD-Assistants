# ラウンドトリップ検証機能設計書

更新日: 2026-04-12

## 1. 概要

設計文書からコードを生成し、そのコードからリファレンス（日本語）を生成する。
設計文書（入力）とリファレンス（出力）を突き合わせ、矛盾がなければ実装は正しい。

設計文書とコードは**デジタルツイン**の関係にあり（→ [設計文書標準 §1](../../design-doc-standard.md)）、表現型が異なるだけで記述される内容は一致しているべき。この前提があるからこそ、日本語同士の照合で設計意図の逸脱を検出できる。

```
設計文書(日本語) → コード(コンパイル) → リファレンス(日本語)
    ↓                                        ↓
    └──── オーケストレーターが突き合わせ ────┘
              矛盾なし         → complete_chunk
              実装が仕様を下回る → Investigation Agent で仕分け → 差し戻し
              実装が仕様を上回る → 設計文書の改訂を人に提案
```

照合は**双方向に働く**。実装が仕様を下回る場合（欠落・矛盾）は差し戻しだが、実装が仕様を上回る場合（書かれた時点で気づかなかった要件やエッジケースが実装中に浮かび上がったケース）は、設計文書を更新する方向で人に相談する。これにより設計書が活文書（living document）として育つ。

これは CDD の「コードを見て考えるな、日本語を見て考えろ」の実践。
コードの中身を読まなくても、**日本語同士の照合**で設計意図の逸脱も進化も検出できる。

## 2. 構成要素

### 2.1 バウンダリ（外部との接点）

- **設計文書** — チャンクの `source_docs` に対応する Markdown
- **リファレンス** — Impl Agent が実装後に生成する日本語文書
- **コード規約資産** — `recipe.json` の `coding_standards` が指す AGENTS.md / linter 設定等（規約適合性の確認に使用）
- **照合結果ファイル** — `docs/ref/verification-{chunk_id}.md`

### 2.2 エンティティ（扱うデータ）

- **乖離（Divergence）** — 設計とリファレンスの差異。重み（致命的/要更新/軽微）付き
- **照合結果** — 判定（OK/NG/要更新）+ 乖離一覧 + サマリー

### 2.3 コントローラー（主要な処理）

- **照合エンジン** — 設計文書とリファレンスを読み比べ、乖離を検出・分類する
- **Investigation Agent** — 照合 NG 時に原因を「実装の問題 / 設計の曖昧さ / テスト不足」に仕分け、差し戻し先を決定する
- **収束制御** — リトライ回数管理、軽微指摘の閾値緩和
- **差し戻し判断** — 致命的→Investigation 経由で分岐、軽微→警告のみ通過、実装が仕様を上回る→設計文書の改訂提案

## 3. リファレンスの位置づけ

### 作成方針

リファレンスは **実装したコードだけを見て作成する** — 設計文書やテストは参照しない。設計文書の文言がリファレンスに混ざると、設計文書 vs リファレンスの照合が自己参照的になり、乖離を検出できなくなる（ラウンドトリップ検証の前提が崩れる）。

**「こうだろう」ではなく「こうなっている」で書く。** 推測ではなく事実だけを記述する。

### 生成の担当とタイミング

Impl Agent が Green フェーズの最後、実装と同じセッションで生成する。テストは Test Agent が別セッションで書いているため、Impl Agent にとってリファレンスの元データは「自分が書いた実装コード」だけになり、作成方針が自然に成立する。

### フォーマットと含める項目

フォーマットは固定しない（フロントエンド / バックエンド / CLI 等で異なる）。含める項目:

1. モジュール構成の概要 — ファイル構成と各モジュールの役割
2. 公開インターフェース — 関数シグネチャ、入力型・出力型
3. 実装ロジック — 処理の流れ、使用しているアルゴリズムやヒューリスティクス
4. 型定義 — 主要な型とその関係

### 用途

1. **オーケストレーターによる設計照合** — 設計文書との矛盾を検出する（主目的）
2. **人のメンテナンス参照** — 後日コードに手を入れる際の手がかり

## 4. レシピへの反映

各チャンクに `reference_doc` フィールドを追加。リファレンスの出力先パスを指定する。

```json
{
  "id": "chunk-01",
  "name": "データベーススキーマ",
  "expected_outputs": ["src/db/schema.sql", "src/db/connection.ts", "tests/db/schema.test.ts"],
  "reference_doc": "docs/ref/chunk-01-db-schema.md"
}
```

`reference_doc` は `expected_outputs` には含めない（リファレンスは実装成果物ではなく検証用文書）。
ただし `complete_chunk` の前にオーケストレーターがリファレンスの存在と内容を確認する。

## 5. 照合の観点と重み分類

オーケストレーターが設計文書とリファレンスを読み比べる際の観点。オーケストレーション手順の全体は [実行フロー §4](execution-flow.md) を参照。

| 重み | 分類 | 例 | 判定 |
|------|------|-----|------|
| 致命的 | 機能の欠落 | 設計に書かれた処理が実装にない | 即差し戻し |
| 致命的 | 型の不一致 | 入出力の型が設計と異なる | 即差し戻し |
| 致命的 | ロジックの矛盾 | 設計と逆の条件分岐をしている | 即差し戻し |
| 要更新 | 設計の進化 | 実装が設計を超えて改善している | 設計文書を更新 |
| 要更新 | 出力スキーマ差異 | 実装にあるフィールドが設計書にない | 設計文書を更新 |
| 軽微 | 命名の揺れ | 設計と微妙に異なる変数名 | 警告のみ |
| 軽微 | 説明の粒度 | リファレンスの方が詳細 | 通過 |

**照合結果の出力:** `docs/ref/verification-{chunk_id}.md` に乖離一覧・判定・サマリーを記録する。出力フォーマットの詳細は [実行フロー §4.4](execution-flow.md) を参照。

## 6. 照合の収束制御

照合→差し戻しのループが終息しない場合に備え、以下の制御を設ける:

- **判断基準の重み付け** — 致命的な乖離は即差し戻し、軽微な乖離は警告に留めて通過させる。リトライのたびに軽微な指摘の閾値を緩和し、収束を促す
- **最大リトライ回数** — 規定回数（デフォルト3回）を超えたら自動差し戻しを停止し、人に判断を仰ぐ
- **収束しないこと自体がシグナル** — リトライ超過は設計文書の記述が曖昧である可能性を示す。「Planner に戻って設計を詰め直す」という判断材料になる

## 7. 設計判断

### なぜ日本語同士で照合するか

コードを直接読んで設計と比較するより、コードから生成した日本語（リファレンス）と設計文書（日本語）を比較するほうが、オーケストレーターの判断精度が高い。これは CDD の「コードを見て考えるな、日本語を見て考えろ」の具体的な実装。

### なぜ双方向で照合するか

一方向（設計 → 実装の追従）だけだと、実装中に浮かび上がった新しい要件やエッジケースを拾えず、設計書が陳腐化する。**実装が仕様を上回るケースを検出して設計書の改訂を提案する**ことで、設計書を活文書として維持できる。Builder は「仕様追従ツール」ではなく、「仕様を洗練するループの一部」として機能する。

### なぜ Investigation Agent を分離するか

乖離の原因は実装とは限らない。「設計の曖昧さ」や「テスト不足」が原因の場合に Impl Agent に差し戻すと、実装側が「直しようのない問題」を延々と回すデッドロックが発生する。原因仕分けを専門エージェントに任せることで:
- 差し戻し先の誤判断を減らす
- 「設計に戻るべき問題」を人にエスカレーションできる
- リトライ収束を早められる

### なぜ収束制御が必要か

無限ループ防止だけでなく、「収束しないこと」が設計の曖昧さを示すシグナルとして機能する。Investigation Agent と組み合わせると、「収束しない → Investigation が繰り返し『設計の曖昧さ』と判定 → 人に返す」という健全なエスカレーションが成立する。

### なぜトークンコストを払ってでもラウンドトリップ検証をするか

リファレンス生成 + 照合 + Investigation は、素直にコードを書くだけの開発に比べて明らかにトークンを消費する。それでもこの仕組みを敷く理由は、コストを上回るメリットが 5 つあるため。

1. **テスト PASS だけでは拾えない乖離を検出する** — Dual-Agent TDD で共有バイアスは排除できるが、命名の揺れ・エッジケースの抜け・暗黙の前提のズレは「テスト通過でも設計と違う」状態を作る。日本語同士の照合はそこを拾う最後の網になる
2. **後段チャンクへの伝播コストを抑える** — chunk-05 の乖離を見逃すと chunk-06 以降はその前提で積み上がる。検証トークンは線形に増えるだけだが、見逃し時のやり直しコストは依存グラフに沿って指数的に膨らむ。早期検出が割に合う
3. **設計文書が活文書として維持される** — 双方向照合で「実装が仕様を上回る」パターンを拾うため、設計文書が実装と一緒に育つ。これがないと docs/ は数ヶ月で実態と乖離し、次の改修で読む価値がなくなる
4. **人のレビュー負荷を下げる** — 「正常時は寡黙」な Issue 更新方針（§5.2）と組み合わせると、人は NG ラベルが付いた Issue だけ読めばよくなる。全チャンクのコードを目で追う作業がなくなる
5. **設計の曖昧さがシグナルとして浮かび上がる** — リトライ超過 = Planner に戻るべき、という判断が自動で出てくる。検証なしだと「なんか進まない」で終わり、原因が設計なのか実装なのか切り分けられない

総じて、**検証にかかるトークンコスト（線形）< 見逃し・手戻り・文書腐敗の総コスト（指数的）** という賭けに立っている。チャンク数が多い大規模プロジェクトほど、この賭けの期待値は大きくなる。

## 8. 検証方針

- 照合の重み分類が適切か（実際の乖離事例で検証）
- Investigation Agent の原因仕分け精度（意図的に設計を曖昧にしたケース、意図的にテスト不足にしたケースで正しく判定できるか）
- 双方向性の動作確認（実装が仕様を上回るケースで、正しく「要更新」判定が出るか）
- 収束制御の閾値が実用的か（CDD-Ghost での実行結果で調整）
- リファレンスの品質がオーケストレーターの照合に十分か
