# チャンク分割機能設計書

更新日: 2026-04-15

## 1. 概要

[設計文書分析](design-analysis.md) の出力を受け取り、**LLM が実装可能な粒度のチャンク**に分割する。依存グラフからトポロジカル順序を決め、レイヤーごとに分割し、合流点には統合テストチャンクを自動挿入する。出力は [実行フロー](execution-flow.md) がそのまま実行できる実行順序付きレシピ。

`split_chunks` が分割を、`export_recipe` が設計文書の該当セクションを埋め込んで最終形の `recipe.json` に変換する。

## 2. 構成要素

### 2.1 バウンダリ（外部との接点）

- **analyze_design の出力** — 依存グラフ・レイヤー分類・推定トークン数
- **recipe.json** — 分割結果の最終成果物（実行エンジンが読む）
- **既存の生成コード** — 前チャンクで生成されたファイル（プレースホルダ解決の対象）

### 2.2 エンティティ（扱うデータ）

- **DraftChunk** — `split_chunks` 出力の中間型。設計文書の内容はまだ埋め込まれていない
- **Chunk** — `export_recipe` で設計文書の該当セクションを埋め込んだ最終形
- **依存グラフ** — チャンク間の実行順序を決定する DAG（Directed Acyclic Graph: 有向非巡回グラフ）
- **execution_order** — DAG のレベル順。同一レベルは並列実行可能
- **統合テストチャンク** — 合流点・レイヤー境界・最終位置に自動挿入されるチャンク

### 2.3 コントローラー（主要な処理）

- **トポロジカルソート** — 依存グラフから実行順序を決定
- **レイヤーマッピング** — 設計レイヤー → 実装レイヤーへの変換
- **チャンク化** — 各レイヤー内で凝集した機能単位に分割
- **統合テスト挿入** — 合流点・境界を検出し、テストチャンクを挿入
- **プレースホルダ埋め込み** — 既存コード参照を `{{file:...}}` 構文で埋め込む
- **設計文書セクション埋め込み** — `export_recipe` 時に `source_docs` の該当セクションを抽出して載せる

## 3. 分割の原則

### 3.1 サイズ制約

| 項目 | 制約 | 根拠 |
|------|------|------|
| 参照設計文書 | 1〜2 本 | 3 本以上で注意散漫・整合性低下 |
| 入力トークン | 8k 以内 | 設計文書 + 既存コード参照 + プロンプト |
| 出力ファイル数 | 3〜5 本 | 多すぎるとファイル間整合性が崩れる |
| 完了条件 | テスト可能 | 次チャンクの前提を保証する |

### 3.2 分割の判断基準

**分割すべき場合:**
- 設計文書の異なるセクションが独立した機能を定義している
- 1 チャンクの推定出力が 12k トークンを超える
- 異なるレイヤー（DB / ロジック / API）にまたがる

**分割すべきでない場合:**
- 2 つの機能が同じテーブル・同じモジュールを密に共有する
- 分割すると片方のチャンクが小さすぎる（< 1k 出力）
- 分割するとチャンク間のインターフェース定義が必要になり、かえって複雑になる

## 4. 分割アルゴリズム

```
Step 0: status による入力選別
         → status: draft / archived の文書を除外
         → 除外した文書は draft_docs として警告（処理は続行）

Step 1: 依存グラフをトポロジカルソート
         → 被依存が多い文書から実装順序を決定（ボトムアップ）

Step 2: 実装レイヤーへマッピング
         設計レイヤー          実装レイヤー
         ──────────          ──────────
         foundation     →    データ層（DB, スキーマ）
         specification  →    ビジネスロジック層
         usecase        →    （実装には直接使わない。検証基準として参照）
         interface      →    API / CLI 層
         operation      →    （テスト・ベンチマーク用）
         context        →    （実装対象外）

Step 3: 各レイヤー内でチャンクに分割
         分割基準:
         - 1 チャンク = 1 つの凝集した機能単位
         - 参照する設計文書は 1〜2 本
         - 生成するファイルは 3〜5 本
         - 完了条件がテスト可能

Step 4: 統合テストチャンクを挿入（§5）

Step 5: execution_order を組み立てる
         → DAG のレベルごとに配列化し、同一レベルは並列実行可能
```

出力は `DraftChunk` の配列と `execution_order`。`implementation_prompt_template` にはプレースホルダを残したまま、`export_recipe` で確定させる。

`split_chunks` は機械的に決まる範囲のみを埋めるため、`expected_outputs` や `completion_criteria` の具体値は `needs_review: true` とともに人のレビュー待ちで返す。

**status による入力選別の詳細:**

| `status` | 扱い | 補足 |
|---|---|---|
| `draft` | 除外 + 警告 | 未完成の文書を後段に流さない |
| `in_review` | 含める | Planner の壁打ちが一段落した状態 |
| `complete` | 含める | 人間承認済み |
| `archived` | 除外（警告なし） | 意図的に廃止された文書 |

フロントマター補完で `in_review` と判定された外部文書もそのままチャンク化される（[設計文書分析 §5.3](design-analysis.md) 参照）。

## 5. 統合テストチャンクの自動挿入

Dual-Agent TDD は各チャンクの単体テストをカバーするが、チャンク間の接続はそれだけでは検証されない。依存グラフを解析し、以下の位置に統合テストチャンクを自動で挿入する。

| 条件 | 挿入位置 | テスト内容 |
|------|---------|-----------|
| 依存グラフで 3 チャンク以上が合流するノード | 合流ノードの直後 | 合流元チャンクの出力が正しく連携するか |
| レイヤー境界（データ層 → ロジック層 → API 層）をまたぐ箇所 | 境界の直後 | 下位レイヤーの出力を上位レイヤーが正しく消費するか |
| 全チャンク完了後 | 最終チャンクの後 | E2E テスト（主要ユースケースの一気通貫実行） |

統合テストチャンクは `test_requirements` のみで構成され、`implementation_prompt` を持たない。Test Agent がテストを生成し、既存の実装に対して実行する（Red フェーズはスキップ）。

## 6. 既存コード参照（プレースホルダ）

チャンク 02 以降は、前チャンクで生成されたコードを参照する必要がある。レシピには `{{file:src/db/schema.ts}}` のようなプレースホルダを埋め込んでおき、実行時に [next_chunks](../3-details/mcp-tools.md) が実際のコードに置換する。

```
DraftChunk（split_chunks の出力）
  ├── implementation_prompt_template  … {source_content} プレースホルダを含む
  ├── test_requirements               … 設計文書から抽出したテスト観点
  └── source_content なし            … まだ設計文書の内容は埋め込まれていない

        ↓ export_recipe で変換

Chunk（recipe.json の最終形）
  ├── implementation_prompt           … {{file:...}} 以外は解決済み
  ├── test_requirements               … そのまま引き継ぎ
  └── source_content                  … 設計文書の該当セクションが埋め込み済み

        ↓ next_chunks で実行時解決

実装指示
  └── {{file:...}} → 前チャンクで生成されたファイルの実際の内容
```

**なぜ 2 段階に分けるか:** `export_recipe` の時点では設計文書の内容は確定しているが、実装コードはまだ存在しない。実行時解決を併用することで、レシピ自体は実装物に依存せず再利用できる。

## 7. 設計判断

### なぜボトムアップで分割するか

被依存が多い文書（foundation 層のデータスキーマ等）から実装すると、後続チャンクは既存の型・テーブル定義を前提にできる。トップダウンだと、上位のインターフェースが下位の実装詳細に依存する形になり、チャンク間のインターフェース定義を毎回決め直すコストが発生する。

### なぜ統合テストを別チャンクにするか

単体テストチャンクの `test_requirements` に統合観点を混ぜると、依存チャンクが全て完了するまで実行できないためチャンク完了判定が遅れる。別チャンクに切り出せば、単体テストは完了させつつ、依存が揃ったタイミングで統合テストチャンクだけ実行できる。

### なぜプレースホルダ方式を使うか

前チャンクのコードをレシピに直書きすると、実装が進むたびにレシピを書き換える必要が出る。プレースホルダ構文で「参照する場所」だけを記録し、実行時に現物を差し込むことで、レシピは実装物と独立に保てる。再実行・部分再実行も容易になる。

### なぜ `expected_outputs` を人レビュー待ちにするか

ファイルパスの具体値は設計文書から機械的に決まらない（設計は「DB スキーマを実装」までしか書かれていない）。推測で埋めると誤ったパスが伝播するので、人が一度目を通すことを前提に `needs_review: true` で返す。

## 8. 検証方針

- 依存グラフの循環を検出・警告できるか
- チャンクサイズがトークン制約の範囲内に収まるか（超えたら警告）
- 統合テストチャンクが合流点・レイヤー境界に正しく挿入されるか
- プレースホルダ構文が `export_recipe` → `next_chunks` で一貫して解決されるか
- `execution_order` が依存関係を反映した並列実行可能な構造になっているか
