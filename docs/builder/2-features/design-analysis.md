# 設計文書分析機能設計書

更新日: 2026-04-15

## 1. 概要

Builder が実装に入る前段として、Planner が完成させた設計文書群を読み取り、**構造・依存関係・技術スタック・コード規約・ドリフト**を抽出する。後続の [チャンク分割](chunk-splitting.md) が機械的な処理だけで動けるよう、設計文書から必要な情報を一度に吸い上げる役割を担う。

`analyze_design` が単独でこの機能を実現する。本体処理（構造解析・依存グラフ構築・規約検出）は LLM を呼び出さず決定論的に動作するが、**入力の前処理としてフロントマターが欠落した文書の補完のみ LLM を使う**（§5 参照）。補完結果はファイルに書き戻すため、2 回目以降は決定論的処理だけで動く。

## 2. 構成要素

### 2.1 バウンダリ（外部との接点）

- **設計文書群** — Markdown ファイル（フロントマター必須、欠落時は LLM で補完する）
- **project_dir** — プロジェクトルート。存在する場合のみ追加ファイルを探索
- **決定ログ** — `decisions.jsonl`（Planner の `track_decision` 出力）
- **既存リファレンス** — `docs/ref/*.md`（ドリフト検出の基準）
- **コード規約資産** — `AGENTS.md` / linter 設定 / `package.json` scripts 等（詳細は [コード規約](coding-standards.md)）

### 2.2 エンティティ（扱うデータ）

- **文書メタデータ** — `path`, `lines`, `layer`, `sections`, `references_to`, `referenced_by`
- **依存グラフ** — 文書間の参照関係を表す DAG
- **レイヤー分類** — foundation / specification / usecase / interface / execution / context
- **技術スタック** — 言語・ランタイム・DB・テストフレームワーク・プラットフォーム
- **コード規約** — 検出された規約ファイルとコマンド定義
- **ドリフト警告** — リファレンス生成後のコード変更履歴

### 2.3 コントローラー（主要な処理）

- **ドリフト検出** — 既存リファレンス以降の git 履歴から、設計と実装の乖離可能性を警告
- **フロントマター補完** — 欠落している文書のみ LLM で `status` / `layer` を推定し、対象ファイルに書き戻す（§5）
- **メタデータ抽出** — フロントマター（補完済みを含む）から `status` / `layer` / `decisions` / `open_questions` を取り込む
- **依存グラフ構築** — Markdown リンクとセクション参照から DAG を組み立てる
- **レイヤー分類** — フロントマターの `layer` を優先、なければ文書内容から推定
- **技術スタック抽出** — 設計文書から言語・フレームワーク等を拾う
- **コード規約検出** — プロジェクトに存在する規約資産を優先度順に収集
- **トークン推定** — tiktoken (cl100k_base) で各文書の推定トークン数を計算

## 3. 処理フロー

```
Step 1: フロントマター補完（欠落している文書のみ）
         → 本文を LLM に渡して status / layer を推定
         → 推定結果をフロントマターとして対象ファイルに書き戻す
         → 補完した文書は frontmatter_autofilled として記録

Step 2: ドリフト検出（project_dir 指定時のみ）
         → docs/ref/ のリファレンスファイルを列挙
         → 各リファレンスの生成日時以降の git コミットを取得
         → 変更ファイルとリファレンスの機能範囲を照合
         → 乖離があれば drift_warnings に記録（処理は止めない）

Step 3: 各文書のメタデータ抽出
         → フロントマター（status, layer, decisions, open_questions）を読み取る
         → Step 1 で補完した文書もここでは通常の文書と同様に扱う

Step 4: 依存グラフ構築
         → Markdown リンクとセクション参照から DAG
         → decisions.jsonl があれば影響文書の関係を反映

Step 5: レイヤー分類
         → フロントマターの layer をそのまま使う

Step 6: 技術スタック抽出
         → 設計文書から言語・DB・テストフレームワーク等を拾う
         → recipe.json の tech_stack に反映

Step 7: コード規約検出
         → 詳細は [コード規約](coding-standards.md) §3 参照

Step 8: トークン推定
         → 各文書の推定トークン数を算出し、チャンク分割の入力とする
```

## 4. ドリフト検出

既存実装がある状態で設計文書を更新してから `analyze_design` を再実行する場合、**リファレンスが生成された時点の実装**と**現在の実装**の間に乖離がないか確認する。

| 観点 | 内容 |
|---|---|
| 基準 | `docs/ref/*.md` の生成日時（フロントマターまたはファイルの更新日時） |
| 対象 | 基準以降のコミットで変更されたファイル |
| 照合 | 変更ファイルがどのリファレンスの機能範囲に属するかをパス対応で判定 |
| 出力 | `drift_warnings` 配列（警告のみ、処理は続行） |

**ドリフトが発生する典型パターン:**

- 設計文書を更新せずに実装を手で修正した（設計 → 実装の一方向が破れている）
- リファレンス生成後に別セッションで実装を拡張した
- git 履歴に残らないファイル変更（gitignore 内のファイル）

ドリフト警告が出た場合、続行するかどうかは人が判断する。無視して進むと、新しいチャンクの実装が古い設計文書に基づいて生成され、実装との整合が崩れる。

## 5. フロントマター補完

[設計文書標準 §8](../../design-doc-standard.md) で設計文書のフロントマターは必須と定めているが、CDD の外から持ち込まれた手書き文書ではフロントマターが欠落している場合がある。`analyze_design` はこれを検出し、**LLM で内容を類推してフロントマターを付記する**。

**重要: 補完はファイル書き戻しを伴う。** `analyze_design` は通常、設計文書の内容を書き換えない。フロントマター補完だけが唯一の例外で、**`docs/` 配下の対象ファイルを直接変更する**。補完後は git diff で変更内容を確認でき、意図しない値なら手で修正するか git から戻せる。フロントマターはメタデータであり仕様そのものではないため、[基本設計 §2.2](../basic-design.md) の「仕様の改訂を独断しない」原則には反しない。

### 5.1 補完の流れ

```
Step 1: 対象文書の先頭を読み、フロントマター（--- で囲まれた YAML ブロック）の有無を判定
Step 2: 欠落している文書のみ LLM に以下を渡して推定:
         - 入力範囲: 文書の先頭 2,000 トークン、または最初の H1 + 概要セクションまで
           （status / layer の推定には十分な情報量で、長文コストを抑える）
         - 推定項目:
           - status: `draft` / `in_review` の 2 値のみ（§5.2 参照）
           - layer: §8.4 の対応表に照らして判定（基本設計なら foundation 等）
Step 3: 推定結果を enum（§5.4）と照合
         → 範囲外なら §5.5 フォールバック
Step 4: 妥当ならフロントマターとして対象ファイルの先頭に書き戻す
Step 5: 分析結果に frontmatter_autofilled として補完した文書を列挙
```

書き戻しはファイルを直接変更するため、Builder の実行後に git diff で内容を確認できる。

### 5.2 補完の限定スコープ

補完対象は `status` と `layer` のみ。`decisions` / `open_questions` は Planner の `track_decision` が作り出す情報であり、本文から類推しても意味のある値にならないため補完しない。

#### status の推定は 2 値に限定

| 推定値 | 条件 |
|---|---|
| `in_review` | 主要セクション（階層ごとのテンプレート項目）が揃っていて、TODO / TBD / 未定 が残っていない |
| `draft` | 上記を満たさない。主要セクションの欠落や「あとで書く」的な空白がある |

- `complete` / `archived` は **LLM が付けない**。[設計文書標準 §8.3](../../design-doc-standard.md) で「人間の明示的承認が必須」と定められているため
- 判断に迷ったら `draft` にフォールバック（未確認文書がチャンク分割に流れ込むリスクを回避）

### 5.3 in_review 以上の文書がチャンク分割の対象

`analyze_design` は全文書のメタデータを返すが、後続の `split_chunks` は **`status: draft` の文書をチャンク化の入力から除外する**。補完で `in_review` が付いた文書は、手書きフロントマターの `in_review` と同じ扱いでチャンク化される。

| `status` | チャンク分割の扱い |
|---|---|
| `draft` | 除外。`draft_docs` として警告を出す |
| `in_review` | 含める |
| `complete` | 含める |
| `archived` | 除外。警告なし（意図的に廃止されたもの） |

これにより、補完結果が `in_review` になった外部文書も通常の Builder フローに載る。Planner を通さずに Builder だけで完結する運用が成り立つ。

### 5.4 許容値（enum）

| フィールド | 標準の許容値 | 補完で LLM が出力してよい値 |
|---|---|---|
| `status` | `draft` / `in_review` / `complete` / `archived` | `draft` / `in_review` のみ |
| `layer` | `foundation` / `specification` / `usecase` / `interface` / `operation` / `context` | すべて |

LLM が許容値以外の文字列を返した場合、または JSON として解釈できなかった場合は §5.5 フォールバックに落とす。許容値との突合は完全一致（大文字小文字を区別しない）で判定する。

### 5.5 フォールバック

LLM 呼び出しに失敗した場合、または推定結果が許容値の範囲外だった場合:

- `status: draft` / `layer: context` を付与して続行（`context` は安全側の分類で、チャンク分割の対象外）
- 警告を `frontmatter_autofill_failed` として記録し、ファイルパスと失敗理由を出力
- 書き戻しは通常どおり実施する（人が後で正しい値に書き換える前提）

この二段構えにより、**推定不能でも処理は止まらず、かつ誤った値がそのまま後段に流れ込まない**。

## 6. 設計判断

### なぜフロントマターを必須にするか

本文からのレイヤー推定は揺れが大きく、同じ内容でも表現次第で分類が変わる。フロントマターが揃っていれば、Builder の分析は常に同じ結果を返す。`status` / `layer` の 2 行を書く小さなコストで、後段のチャンク分割の安定性が大きく向上する（[設計文書標準 §8](../../design-doc-standard.md) 参照）。

### なぜフロントマター補完だけ LLM を使うか

レシピエンジンは LLM に依存しないというアーキテクチャ原則（[基本設計 §2.1](../basic-design.md)）がある。決定論的に動かないと「同じ文書を渡したのに違うチャンクが出る」再現性問題が発生するため。

ただしフロントマター補完は**入力の前処理**であり、レシピエンジン本体の処理ではない。一度書き戻せば次回以降は決定論的処理だけで動くため、再現性問題は初回のみに限定される。このため、**入力の整備に限って LLM の利用を許す**立て付けにしている。

### なぜ書き戻しまでやるか

補完結果をメモリ上だけで使うと、次回 `analyze_design` を走らせるたびに LLM 推定が走り、そのたびに違う結果が出るリスクが残る。ファイルに書き戻せば、初回補完後は手書きのフロントマターと同じ扱いになる。人が書き戻された内容を見て修正することもできる。

### なぜ status 推定を 2 値に限定するか

[設計文書標準 §8.3](../../design-doc-standard.md) で `in_review → complete` は「人間の明示的承認が必須」とされている。LLM が外部文書を見て「完成してる」と判定して勝手に `complete` を付けると、**人間の承認なしにチャンク分割の入力として確定してしまう**。

一方で、一律 `draft` にすると実装可能な精度の外部文書でもチャンク分割から除外され、Planner なしに Builder を使う運用が成り立たなくなる。2 値サポート（`draft` / `in_review`）で、**LLM の判断を承認の手前まで**に閉じ込めつつ、**Planner を通さない運用も可能**にする。迷ったら `draft` フォールバックで、未確認文書が `in_review` 扱いで流れ込むリスクも抑える。

### なぜドリフト検出を分析の最初に置くか

古い設計文書を信じて分割・実装を進めると、後続チャンクが破綻した設計の上に積み上がる。分析の入口で警告を出すことで、「設計を更新すべきか、実装を捨てて作り直すか、ドリフトを許容するか」の判断を人に早期に求められる。

### なぜ技術スタックを設計文書から抽出するか

チャンクに「TypeScript で実装」と書き込むのではなく、recipe.json の `tech_stack` に一度集約する。これにより、技術スタックが変わったとき（Python → Rust 等）に設計文書とレシピの両方を書き換える二重管理を避けられる。

## 7. 検証方針

- フロントマターがある文書は常に同じレイヤー分類になるか（決定論性）
- フロントマターが欠落した文書で LLM 補完が正しいレイヤーを推定するか
- status 推定が `draft` / `in_review` の 2 値に限定されているか（`complete` / `archived` を出力しないか）
- 判断に迷うケースで `draft` フォールバックが働くか
- 補完後のファイル書き戻しで元の本文が破損しないか
- LLM 呼び出し失敗時に `draft` / `context` フォールバックが働くか
- LLM が許容値外の値を返したときに enum チェックで弾かれ、フォールバックに落ちるか
- LLM に渡す入力範囲（先頭 2,000 トークン）が status / layer 推定に十分か（精度を実文書で計測）
- `split_chunks` が `status: draft` の文書を除外し、`draft_docs` として警告するか
- 循環参照を含む依存グラフを検出・警告できるか
- ドリフト検出が既存のリファレンス生成日時を正しく基準にできるか
- 規約ファイルが複数存在する場合の優先順位が期待どおり働くか
- decisions.jsonl の影響文書が依存グラフに反映されているか
