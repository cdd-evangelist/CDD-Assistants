## v0.2（次期）

### コンシェルジュの独立

clarify_idea 内に埋め込まれているルーティングロジックを独立したモジュールに分離する。

```
ユーザーの入力
    ↓
コンシェルジュ（独立モジュール）
    ├── ワンショット → 直接実装
    ├── フル CDD   → Planner
    └── メンテナンス → Maintainer（新設）
```

ルーティング先が増えるタイミングで分離するのが自然。

### Planner にスキャフォールド機能を追加

Planner が壁打ちのフェーズ進行に合わせて、design-doc-standard.md §5.1 の標準フォルダ構成（`docs/`, `1-usecases/`, `2-features/`, `3-details/`, `4-ref/`）を自動生成する。

**論点（実装前に決める）:**
- どのツールがトリガーになるか（`track_decision` / 新ツール `scaffold_phase` / `clarify_idea` の route 判定時）
- フェーズ移行の判定基準（基本設計 → 機能設計 → 詳細設計）
- 既存フォルダがある場合の扱い（上書きしない / 追記のみ / 確認プロンプト）
- README.md / basic-design.md のテンプレート挿入を含めるか

現状は人（または Claude）が手動でフォルダを作る運用。dogfooding でこの不便さが顕在化したタイミングで設計を確定させる。

### Maintainer の新設

設計・実装済みリポジトリの部分修正に特化したアシスタント。

**解決する問題:**
- 既存リポジトリに対して Builder を最初から通すと、変更のない部分まで再実行して無駄が多い
- ドリフト検出は「警告するだけ」で、修正アクションにつながらない

**想定する機能:**
- 設計文書の変更差分から影響チャンクを特定
- 既存の実装コードを `done` として扱い、影響範囲だけ再実行
- リファレンスが古いチャンクだけ再検証（差分ラウンドトリップ）
- Builder のドリフト検出結果を入力として受け取れる

**コンシェルジュとの連携:**
- 「この部分だけ直して」「設計変わったから反映して」→ コンシェルジュが Maintainer にルーティング
