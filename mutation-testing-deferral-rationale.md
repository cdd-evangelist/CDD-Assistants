## 3. 補足論点（assistant が追加した主張）

agent の薄い根拠を補うために、以下の論点を追加で検討する:

### 3.1 「本丸は Dual-Agent TDD + ラウンドトリップ検証」

- Dual-Agent TDD（10）とラウンドトリップ検証（12）が動いていない状態では、Mutation Testing を回す対象（実装されたコードと、それに対応するテスト）の質が未知数
- 土台が不安定なまま高度な品質計測をしても、スコアがノイズ混じりで信頼できない

**反論の余地:** Mutation Testing は Dual-Agent TDD なしでも既存テストに対して動く。Builder 自体の既存テスト（`builder/tests/`）に適用すれば、**ブートストラップの品質ゲート**として機能する可能性はある。

### 3.2 「Mutation Testing は 1→10 フェーズで効く」

- 0→1（機能を動かす）の段階より、1→10（品質を詰める）の段階で効く
- 初期実装の「そもそもテストが薄い／片寄っている」問題は、静的チェック（4項目）だけでも相当カバーできる

**反論の余地:** 「1→10 で効く」は一般論。Builder は AI 生成コードを扱うので、**人間が書くより抜けが特殊**。早い段階で Mutation Testing を入れた方が、AI 生成テストの典型的な抜けパターンを発見できるかもしれない。

### 3.3 「静的チェックは軽い」

- 静的チェック（4項目）は推定 2h の工数で実装可
- Mutation Testing だけ先送りしても、テスト品質保証の半分は v0.1 で提供できる

**反論の余地:** 静的チェックは正規表現や AST ベースの実装で済むが、Mutation Testing は**変異体注入→テスト実行→スコア算出**という独立した実行パイプラインが必要。分離するとフィードバックループ（Survived mutants → Test Agent への追加指示）の設計も分断される。一緒に実装した方が整合性が取れる可能性。

### 3.4 工数・リスクの時間軸評価

- agent 提示の 3-5h（Mutation Testing のみ）は **Stryker 統合だけ**の数字。Builder が将来扱う言語（Python, Go 等）も含めると合計は 10h+
- v0.1 のブートストラップ期に 10h+ 消費するのは、**Dual-Agent TDD（5-8h）と同等以上**の投資
- v0.2 で段階導入すれば、**言語ごとにニーズが出てきてから**入れられる（YAGNI 的な利点）

## 5. ひっくり返す余地

以下のいずれかに該当する場合は **A（未実装）を再考する価値あり**:

1. **Builder の既存テストの質が疑わしい**: 今の `builder/tests/` が甘いなら、Stryker で炙り出してから本体改修に入った方が良い
2. **AI 生成テストの特性上、早期の Mutation Testing が効く確信がある**: 人間と違う抜け方をする等の仮説
3. **Python を v0.1 中に扱う予定**: Python プロジェクトで Builder を使うユースケースが先にあるなら、mutmut も入れる動機が出る
4. **ユーザー（人間）がテスト品質の可視化を早く見たい**: 開発中のモチベーション維持のため

## 6. 最終決定

- **選択肢: A（v0.1 は未実装）**
- **決定日: 2026-04-21**

### 決定理由（別セッションで得た論点が決め手）

1. **v0.1 の主な失敗モードは Mutation Testing のレイヤーと別**
   - v0.1 で恐れるべきは「パイプラインが動くか」「依存解決が合ってるか」「エージェント起動が通るか」
   - Mutation Testing はこれらが安定した後（1→10 フェーズ）で効く道具
   - **§5 の反論「早期投入で AI 生成テストの抜けを炙り出せる」は、そもそも v0.1 の主要リスクではないので優先度が下がる**

2. **Mutation Testing は土台が不安定だとノイズだらけ**
   - テストがまだフラフラしてる段階で変異注入しても、「実装変更が原因か、元々テストが不安定か」区別つかない
   - スコアが信頼できない段階での投入は害

3. **Red 確認 + 静的チェック + ラウンドトリップ照合の 3 層で v0.1 は足りる**
   - Red 確認（テストが FAIL することを先に確認）で Assertion の空打ちは防げる
   - 静的チェック 4 項目（パラメータ網羅・異常系・統合ポイント・Assertion 品質）で明白な欠陥は拾える
   - 残る「比較が甘い」系は、ラウンドトリップ照合（設計文書 vs リファレンスの突き合わせ）で**別方向から**検出できる可能性がある
   - Mutation Testing はこの 3 層の**補強**であって、不在でも壊滅的ではない

### 再検討タイミング

以下のいずれかが観測されたら B（Stryker 統合）を再検討:

- 静的チェック + ラウンドトリップ照合が安定稼働したが、「比較が甘い assert」由来のバグが頻出する
- AI 生成テストの抜けパターンを定量化したい実用動機が出る
- Python プロジェクトで Builder を使う案件が来る（mutmut 検討）
